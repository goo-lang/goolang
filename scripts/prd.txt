# Overview  
Goo is a systems programming language designed as a Go superset that incorporates the best features from Rust (memory safety), Zig (compile-time execution), Erlang (fault tolerance), and nng (advanced messaging patterns). It aims to provide Go's simplicity and readability while adding powerful safety and concurrency features needed for modern systems programming, particularly microkernel development like LughOS.

The language solves the problem of choosing between safety and simplicity in systems programming. Currently, developers must choose between C (unsafe but simple), Rust (safe but complex), Go (simple but garbage-collected), or Zig (powerful but lacking high-level concurrency). Goo provides a unified solution that doesn't force these trade-offs.

# Core Features  

## Go Compatibility
- **What it does**: Accepts any valid Go program without modification
- **Why it's important**: Allows gradual migration and leverages existing Go ecosystem
- **How it works**: Parser recognizes full Go syntax; compiler generates compatible output

## Memory Safety
- **What it does**: Prevents null pointer dereferences, buffer overflows, and use-after-free errors
- **Why it's important**: Eliminates entire classes of security vulnerabilities
- **How it works**: Ownership tracking (simplified from Rust), bounds checking, and nullable types (`?int`)

## Error Unions (Zig-style)
- **What it does**: Provides clean error handling with `!` return types
- **Why it's important**: More ergonomic than Go's `if err != nil` or Rust's `Result<T,E>`
- **How it works**: Functions return `!T` and use `try`/`catch` for propagation

## Advanced Concurrency
- **What it does**: Extends Go's goroutines with Erlang-style actors and supervision
- **Why it's important**: Enables self-healing systems crucial for OS services
- **How it works**: Actors with message passing, supervision trees for automatic restart

## Channel Patterns
- **What it does**: Native support for nng patterns (pub/sub, req/rep, push/pull)
- **Why it's important**: Simplifies building distributed systems and microkernel IPC
- **How it works**: Extended channel syntax: `make(pub chan Message)`

## Compile-Time Execution
- **What it does**: Execute code at compile time for optimization and configuration
- **Why it's important**: Zero-overhead abstractions and kernel configuration
- **How it works**: `comptime` blocks run during compilation, generating optimized code

# Technical Architecture  

## System Components

### Compiler Pipeline
```
Source (.goo) → Lexer → Parser → AST → Type Checker → 
Optimizer → LLVM IR → Native Code
```

### Core Components
1. **Compiler (Go-based)**
   - Lexer: Tokenizes Goo source code
   - Parser: Builds AST supporting Go + Goo syntax
   - Type Checker: Enforces safety with ownership tracking
   - Code Generator: Produces LLVM IR
   - Optimizer: Compile-time execution and optimizations

2. **Runtime (C/Assembly)**
   - Memory Manager: Arena/pool allocators
   - Scheduler: M:N goroutine scheduling
   - Channel System: Lock-free channels with patterns
   - Actor System: Message passing and supervision
   - FFI Layer: C interoperability

3. **Standard Library (Goo)**
   - Core types and traits
   - I/O abstractions
   - Networking (with channel patterns)
   - OS interfaces
   - Collections

### GPU Programming Model
```go
// GPU kernel definition
gpu kernel vec_add(a: []float32, b: []float32, c: mut []float32) {
    tid := gpu.thread_id()
    if tid < len(a) {
        c[tid] = a[tid] + b[tid]
    }
}

// Host-side GPU execution
func compute_on_gpu() !void {
    // GPU memory allocation with ownership
    gpu_a := try gpu.alloc_array(float32, 1000)
    defer gpu_a.free()
    
    gpu_b := try gpu.alloc_array(float32, 1000)
    defer gpu_b.free()
    
    gpu_c := try gpu.alloc_array(float32, 1000)
    defer gpu_c.free()
    
    // Copy data to GPU
    try gpu_a.copy_from(host_array_a)
    try gpu_b.copy_from(host_array_b)
    
    // Launch kernel
    try gpu.launch(vec_add, 
        blocks: (1000 + 255) / 256,
        threads: 256,
        args: (gpu_a, gpu_b, gpu_c))
    
    // Synchronize and copy result back
    try gpu.synchronize()
    try gpu_c.copy_to(host_result)
}

// Advanced GPU patterns
gpu kernel reduce_sum(input: []float32, output: mut []float32) {
    shared data: [256]float32
    tid := gpu.thread_id()
    
    // Load data into shared memory
    data[tid] = if tid < len(input) { input[tid] } else { 0.0 }
    gpu.sync_threads()
    
    // Parallel reduction
    stride := 128
    while stride > 0 {
        if tid < stride {
            data[tid] += data[tid + stride]
        }
        gpu.sync_threads()
        stride /= 2
    }
    
    if tid == 0 {
        output[gpu.block_id()] = data[0]
    }
}
```

### Kernel Services Architecture
```go
// Kernel service definition
kernel service FileSystem: Actor {
    state: FsState
    
    handle CreateFile(path: String, mode: FileMode) -> !FileHandle {
        // Secure file creation with capability checking
        try capability.check(self.caller(), FileCreate, path)
        return try fs.create(path, mode)
    }
    
    handle ReadFile(handle: FileHandle, buffer: mut []u8) -> !usize {
        try capability.check(self.caller(), FileRead, handle)
        return try fs.read(handle, buffer)
    }
}

// Hardware driver framework
driver PciDriver: HardwareDriver {
    device_class: PciClass
    
    fn probe(device: PciDevice) -> bool {
        return device.class == self.device_class
    }
    
    fn init(device: PciDevice) -> !DriverInstance {
        unsafe {
            // Map device memory
            mmio_base := map_device_memory(device.bar0)
            // Initialize device registers
            mmio_write(u32, mmio_base + 0x04, DEVICE_ENABLE)
        }
        return DriverInstance { device, mmio_base }
    }
}
```

# Logical Dependency Chain

## Foundation Layer (Must Build First)
1. **Lexer**: Needed for all parsing
2. **Parser**: Depends on lexer, needed for AST
3. **AST**: Core data structure for entire compiler
4. **Basic Type System**: Required for any code generation
5. **Minimal Runtime**: Required to run any compiled code

## First Visible Output
**Target**: Compile and run "Hello, World" (Week 2-3)
- Lexer for basic tokens → Parser for function declarations → Type checker for primitives → Code generator for print → Link with minimal runtime

This provides immediate feedback and validates the architecture.

## Feature Building Blocks
1. **Error Unions** (builds on type system)
   - Extends type system → Updates parser → Modifies code generation
   - Can compile error-handling code

2. **Null Safety** (builds on type system)
   - Extends type system → Adds type checking rules
   - Can write null-safe code

3. **Ownership** (builds on type system + control flow)
   - Needs CFG analysis → Borrow checker → Move semantics
   - Enables memory safety

4. **Goroutines** (builds on runtime)
   - Needs scheduler → Stack management → Context switching
   - Enables concurrency

5. **Channels** (builds on goroutines + type system)
   - Needs goroutines → Synchronization primitives
   - Enables communication

6. **Actors** (builds on goroutines + channels)
   - Needs message passing → Supervision logic
   - Enables fault tolerance

## Incremental Improvements
Each feature follows this pattern:
1. **Syntax Support**: Add to lexer/parser
2. **Semantic Support**: Add to type checker
3. **Code Generation**: Add to LLVM backend
4. **Runtime Support**: Add to runtime library
5. **Testing**: Add test cases
6. **Documentation**: Update docs

This allows each feature to be fully functional before moving to the next.

# Risks and Mitigations  

## Technical Challenges

### Risk: LLVM Complexity
- **Challenge**: LLVM API is complex and changes between versions
- **Mitigation**: Start with LLVM 17 (stable), create abstraction layer, maintain compatibility layer

### Risk: Ownership Model Complexity
- **Challenge**: Simplified ownership still needs to be correct
- **Mitigation**: Start with basic move semantics, add borrowing gradually, provide clear error messages

### Risk: Runtime Performance
- **Challenge**: M:N scheduling and channels could be slow
- **Mitigation**: Profile early, optimize hot paths, provide unsafe escapes for performance

## MVP Scoping

### Risk: Feature Creep
- **Challenge**: Trying to implement everything at once
- **Mitigation**: Strict phasing, Go compatibility first, add features incrementally

### Risk: Incomplete Go Compatibility
- **Challenge**: Missing Go features block adoption
- **Mitigation**: Prioritize most-used Go features, document limitations, provide migration path

## Resource Constraints

### Risk: Single Developer Bandwidth
- **Challenge**: Large scope for one person
- **Mitigation**: Focus on core features, leverage existing tools (LLVM, parser generators), build community early

### Risk: Testing Complexity
- **Challenge**: Compiler testing is complex
- **Mitigation**: Reuse Go's test suite where possible, start with simple tests, use property-based testing

### Risk: Documentation Burden
- **Challenge**: Docs needed for adoption
- **Mitigation**: Document as you build, use examples as docs, leverage community

## Technical Specifications

### Language Grammar (EBNF subset)
```
Program     = Package Imports Declarations
Package     = "package" Identifier
Import      = "import" StringLit
Function    = "func" Identifier Parameters ["!"] Type Block
Type        = BasicType | "?" Type | ChannelType
ChannelType = ["pub"|"sub"|"req"|"rep"] "chan" Type
```

### Runtime ABI
- C calling convention for FFI
- Goroutine state in thread-local storage
- Channels as opaque pointers
- Actor mailboxes as lock-free queues

### Optimization Levels
- `-O0`: No optimization, fast compilation
- `-O1`: Basic optimizations
- `-O2`: Standard optimizations (default)
- `-O3`: Aggressive optimizations
- `-Osize`: Optimize for size (kernel mode)
