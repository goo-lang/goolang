# Goo v1 Roadmap — evidence-based synthesis

**Date:** 2026-06-29  
**Source:** `/goo-v1-roadmap` workflow (run wf_a531154c-a3b) — 17 agents, all conclusions verified by running `bin/goo`, not trusting status docs.  
**Method:** 10 parallel subsystem investigators → adversarial refutation of 6 roadmap claims → dependency-ordered synthesis.

> This supersedes `docs/10-IMPLEMENTATION-ROADMAP.md` (the aspirational 27-month vision) as the *working* v1 plan. It does not trust the status markdown files or task-master 'done' counts — every claim below was checked against the running compiler.

## v1 Definition

> v1 = a Goo compiler that reliably compiles AND RUNS a useful Go-compatible subset from ANY working directory, with both differentiators working end-to-end in BOTH directions: ?T nullable (already done) and !T error unions including the failure path (error construction, catch firing, try propagation). 'Useful subset' means: functions, structs + value/pointer-receiver methods, control flow with correct break/continue, slices/maps via Go-standard composite literals, string concat, defer, basic fmt (Println + Printf), and a sound enough type checker that signature/arg/return mismatches are rejected at type-check time rather than crashing LLVM. Grand-vision items (self-hosting, microkernel, capability security, formal verification, GPU, WASM, Rust interop, macros, async runtime, package manager) are explicitly post-v1.

## Verified baseline (current real state)

VERIFIED current state (bin/goo built Jun 29, lib/libgoo_runtime.a present): The pipeline lex->parse->typecheck->codegen->link->run WORKS for a real Go subset. Empty main, hello-world, integer/float arithmetic, recursion, structs + field access, VALUE-receiver methods, multiple returns, pointers, maps (map[string]int{} via Goo path), switch, goroutines/channels/select, and comptime-const evaluation all compile and run with correct output. ?T nullable is fully working end-to-end (int/int64/string/struct, if-let, ==nil, auto-wrap). CORRECTION to the framing premise: ASI is NOT a root blocker — the lexer already does targeted ASI (lexer.c:114-135) and the grammar tolerates newline-separated statements, so multi-statement bodies parse and run WITHOUT explicit semicolons (refuted). Likewise 'linking always fails / even empty main fails' is FALSE — it only fails from a non-repo-root cwd due to the relative runtime path (refuted). 'Structs/methods/interfaces have zero support' is FALSE — structs and value-receiver methods work; only interfaces are truly absent (refuted).

VERIFIED real defects blocking a useful/honest v1: (1) Runtime archive path is cwd-relative hardcoded — compiler unusable outside repo root; make install ships a broken toolchain. (2) break/continue are silent no-op stubs (confirmed at statement_codegen.c:116-119) — SEVERE miscompilation: any loop with break/continue runs wrong. (3) !T error unions are success-ONLY: codegen_generate_error_return always hardcodes is_error=false, the error-constructing path is dead code, there is no surface syntax to introduce an error, catch never fires, try emits 'unreachable' — the headline differentiator cannot represent failure and produces garbage at runtime. (4) Type checker has TODO stubs: return-type vs signature (type_checker.c:961), call-site arg count+type (expression_checker.c:595/601) — mismatches slip through to LLVM 'Module verification failed' instead of clean diagnostics. (5) fmt.Println of any non-scalar (struct/slice/?T/!T/pointer) emits INVALID IR that fails the verifier. (6) Go-standard []T{...} composite literals are misparsed as struct literals ('Unknown type int in struct literal'); only Goo-native [1,2,3] works — blocks idiomatic slices/range. (7) Pointer-receiver method on an addressable value miscompiles (by-value call, verifier failure). (8) defer is rejected by the type checker and is a codegen stub. (9) string + concat unsupported despite runtime goo_string_concat existing. (10) Interfaces are entirely unimplemented (parse error; INTERFACE token declared but no grammar rule). (11) fmt.Printf/Sprintf/Errorf unsupported; imports are a no-op (unknown/unused imports not caught); only 4 packages, all hardcoded at 3 hand-synced sites. (12) Grammar has 224 conflicts (68 s/r + 156 r/r, NOT the 142 docs claim), named return params entirely absent, ~13 dead GPU/wasm/js nonterminals. (13) Headline 'make test 100% pass' is 77 in-process C unit tests that NEVER invoke bin/goo; real e2e signal lives in separate, triple-maintained, drifting probe lists. (14) Most 'advanced' subsystems (macros, async, security/taint, optimizer, auto-parallel, package mgr) are linked-but-uncalled or not compiled; only comptime-const is genuinely wired. -O flag is a silent no-op. task-master reports 34/42 done but several 'done' power-features (GPU/WASM/async/auto-parallel) don't even parse — 'done' is not a reliable definition of done.

## Critical path

- CORRECTION: ASI is NOT the keystone. The adversarial verdict refuted 'the lexer discarding newlines is THE root blocker' — multi-statement bodies already parse, type-check, codegen, link, and run without explicit semicolons (lexer.c:114-135 does targeted ASI). ASI work is real but is correctness hardening (Phase 4), not a build blocker.
- TRUE KEYSTONE 1 (P0-1): Resolve the runtime archive path relative to the binary. Until this is fixed the compiler is unusable outside the repo root and `make install` ships a broken toolchain — it gates the out-of-tree smoke test and any honest 'usable compiler' claim.
- TRUE KEYSTONE 2 (P0-2): Implement break/continue codegen. Confirmed silent no-op stub; every loop using break/continue currently miscompiles. This is a correctness floor before any further codegen work can be trusted.
- P0-3/P0-5: Stop invalid-IR emission on non-scalar print and stand up the golden runner — establishes a trustworthy regression net used by every later phase.
- P1-1 -> P1-2 -> P1-3/P1-4/P1-5 -> P1-6: The !T error-union failure path. This is the load-bearing v1 differentiator chain (error intro -> codegen is_error -> return typecheck / catch fires / try propagates -> e2e probe). ?T is already done; !T failure is the gap that makes v1 meaningful.
- P2-1/P2-2: Return-type and call-arg type checking — converts LLVM verifier crashes into clean diagnostics; required for a compiler that 'reliably' compiles and rejects bad input.
- P3-1 -> P3-2: Go-standard []T{...} slice literals (typecheck then codegen) unblock idiomatic slices/range, the most-used missing Go construct.
- Phase 6 interfaces are sequenced LAST among functional work: heaviest (parser+typecheck+codegen, lowest confidence on the vtable task) and not required for the core useful subset or the differentiators.

### Phase 1 — Differentiator completion: !T error unions end-to-end (failure path)

**Goal:** Make the headline differentiator actually represent and handle failure. Today !T is success-only and produces garbage on the error path; this phase wires error construction, catch, try-propagation, and return type-checking so a function can return an error and a caller can handle it.

**Exit gate:** A program where f() returns error("X"), a caller does `r := f() catch e { print(e); default }` prints X (not garbage) and binds e; `try f()` in an !T function propagates the real is_error up to a top-level catch; `try` in a non-!T function is a type error (not silent unreachable); `func f() !int { return "x" }` is rejected at type-check. A single differentiators probe exercising BOTH success+error halves of !T and ?T is green in `make verify` and fails red if any error path regresses to garbage.
