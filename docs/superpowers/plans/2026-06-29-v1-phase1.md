# v1 Phase 1 — `!T` Error Unions: the Failure Path (Implementation Plan)

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:subagent-driven-development. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Make Goo's headline differentiator — `!T` error unions — actually represent and handle **failure**. Today `!T` is success-only: there is no surface syntax to introduce an error, the return path always builds the success variant (`is_error=0`), so `catch` never fires and `try` emits `unreachable`. This phase wires error construction (`error(msg)`), return type-checking, and confirms the (already-present) catch/try dispatch.

**Architecture:** The error-union ABI `{ i1 is_error, union{value, error} }` and both constructors (`codegen_create_error_union_success` / `_error`) already exist, as does the `is_error` dispatch in `codegen_generate_catch_expr_impl` (error_union_codegen.c:188) and `codegen_generate_try_expr_impl` (:136). The **only** missing piece is a way to construct an error value. So the linchpin is the `error(msg)` builtin + the return-path wiring; the rest of the machinery then lights up.

**Tech Stack:** C23, LLVM C API, the Goo compiler, the existing error-union codegen (`src/codegen/error_union_codegen.c`), type checker (`src/types/`), Makefile + golden runner.

**Spec:** `docs/2026-06-29-v1-roadmap.md` (Phase 1). Decision: error-introduction syntax is the **`error(msg)` builtin** (default string error type).

## Global Constraints

- **Build env:** `eval $(opam env --switch=default)` before any `make`; ignore the `parser.tab.c` recipe-override warnings.
- **Gates:** `make verify` (incl. `test-golden`, the P0-5 oracle) and `make test` (76/1) must stay green. Local hooks gate commits/pushes; CI disabled.
- **Error-union ABI is fixed** (`{ i1 is_error, union{value,error} }`) — do NOT change it. The default error value is a **string** (the message); the catch error-var binds to it (type_mapping.c maps the error slot to a goo_string).
- **`error(msg)` design (localized, low-risk):** `error` is a builtin taking one string arg. It is only valid where an `!T` is expected — primarily `return error("...")` inside an `!T` function, and as the RHS feeding `catch`/`try`. Do NOT introduce a general "error marker type" flowing through the whole type system; special-case it at the `!T` return path and validate the string arg. Codegen builds the error variant using the *enclosing function's* error-union LLVM type.
- Commit discipline: conventional commits, `--no-gpg-sign`, stage only named files, never `git add .` / `.superpowers/` / `build/`.
- TDD throughout: RED probe first (golden case or negative probe), confirm it fails for the stated reason, implement, confirm GREEN, run `make verify`.
- `int` is i32 today (the `int→i64` change is a separate milestone) — keep Phase 1 fixtures using small int values so they're width-agnostic.

// erru_fail_probe: an !int function returns error("boom"); the caller's catch

}
