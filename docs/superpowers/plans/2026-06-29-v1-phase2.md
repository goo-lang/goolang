# v1 Phase 2 — Type-Checker Soundness (Implementation Plan)

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:subagent-driven-development. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Replace silent-acceptance TODO stubs so bad programs are rejected at type-check with clear, source-located diagnostics instead of crashing the LLVM verifier with "Module verification failed". A compiler that "reliably compiles" must reject bad input cleanly.

**Architecture:** Two missing checks in the type checker (return-type for non-`!T` functions; call-site arg count + types) plus a pointer-receiver auto-address-of codegen fix and a regression probe for the already-working aggregate-print rejection. All additive — each converts an existing LLVM-verifier crash into a `type_error`.

**Tech Stack:** C23, the Goo type checker (`src/types/`), codegen (`src/codegen/`), Makefile golden runner.

**Spec:** `docs/2026-06-29-v1-roadmap.md` (Phase 2). Note: P1-3 already added `!T` return checking; P0-3 already made `fmt.Println` of unsupported (incl. aggregate `?T`/`!T`) types a clean error.

## Global Constraints

- **Build env:** `eval $(opam env --switch=default)` before any `make`; ignore the `parser.tab.c` recipe-override warnings.
- **Gates:** `make verify` (incl. `test-golden`) and `make test` (76/1) stay green. Local hooks gate commits/pushes; CI disabled.
- **Each task converts a verifier crash into a clean `type_error`** — the negative probe must confirm: non-zero exit AND a source-located diagnostic AND the LLVM verifier is NOT reached (grep stderr must NOT contain "Module verification failed").
- Do NOT over-reject: every currently-passing probe (the 46 golden cases + all `verify` probes) must stay green — valid programs must still compile.
- `int` is i32 today (separate milestone) — keep fixtures small-valued.
- Commit discipline: conventional commits, `--no-gpg-sign`, stage only named files, never `git add .` / `.superpowers/` / `build/`. TDD: RED negative probe first.

### Task 1 (P2-1): Check `return` value type against a non-`!T` function signature

**Files:**
- Modify: `src/types/type_checker.c` (`type_check_return_stmt`, the `{ Type* return_type = ... }` block ~line 1001-1036 — the existing logic only validates the `!T` case; add the non-`!T` case)
- Test: `return-mismatch-probe` (negative)

**Interfaces:** Consumes `checker->current_return_type`.

- [ ] **Step 1: Write the RED negative probe** — `return-mismatch-probe` Makefile target: `func f() int { return "str" }` must fail with a clean type error (non-zero exit, message naming the mismatch), and stderr must NOT contain "Module verification failed".
- [ ] **Step 2: Run → FAIL** — today this reaches the verifier: "Function return type does not match operand type of return inst!".
- [ ] **Step 3: Implement** — in `type_check_return_stmt`, after computing `return_type` for a present value, when `expected` (`current_return_type`) is NOT an error union: require `type_compatible(return_type, expected)`, else `type_error("return type mismatch: cannot return %s from a function returning %s", ...)`. (The `!T` branch already exists; this is the `else`.) Also handle the value-returned-from-void case (`func f() { return 5 }` → error) if not already caught.
- [ ] **Step 4: Run → PASS** — bad return rejected cleanly; `make verify` green (all valid returns still accepted, incl. the `!T` and multi-return probes).
- [ ] **Step 5: Commit** — `feat(types): type-check return value against non-!T function signature (P2-1)`.

### Task 2 (P2-2): Check call-site argument count + types for user functions

**Files:**
- Modify: `src/types/expression_checker.c` (the call-arg TODO stubs ~line 597-608: `arg_count` is computed but unused, "TODO: Check argument type compatibility", "TODO: Check argument count")
- Test: `call-arity-probe` + `call-argtype-probe` (negative)

**Interfaces:** Consumes the callee's function `Type` (params + count). MUST NOT apply to variadic/special builtins (println/print/make_chan/len/cap/append/error) — those are special-cased; only check ordinary user functions with a known param list.

- [ ] **Step 1: Write the RED probes** — (a) `call-arity-probe`: `func add(a int, b int) int {...}; add(1)` → clean "wrong number of arguments" error. (b) `call-argtype-probe`: `add("x", 2)` → clean "argument 1: cannot use string as int" error. Both: non-zero exit, no "Module verification failed".
- [ ] **Step 2: Run → FAIL** — today: "Incorrect number of arguments passed to called function!" / parameter-type verifier crash.
- [ ] **Step 3: Implement** — in the call type-check, when the callee resolves to a user function type with a known param list (skip variadic builtins): compare arg count to param count (`type_error` on mismatch); for each arg, `type_compatible(arg_type, param_type)` (`type_error` naming the position on mismatch). Use the existing `arg_count` loop.
- [ ] **Step 4: Run → PASS** — both probes rejected cleanly; `make verify` green (every existing call — methods, multi-arg goroutines, builtins — still type-checks).
- [ ] **Step 5: Commit** — `feat(types): check call argument count + types for user functions (P2-2)`.

### Task 3 (P2-3): Auto-address-of for pointer-receiver method calls on addressable values

**Files:**
- Modify: `src/codegen/` (the method-call codegen — find where a `p.method()` call passes the receiver; investigate `call_codegen.c` / `composite_codegen.c` method dispatch)
- Test: `examples/ptr_recv_probe.goo` (+ expected) — golden

**Interfaces:** Consumes the method's receiver kind (value vs pointer) from the resolved method type.

- [ ] **Step 1: Investigate + write the RED golden probe** — `examples/ptr_recv_probe.goo`: a struct with a pointer-receiver method (`func (p *Counter) inc()` mutating a field), called on an addressable local `c.inc()`. Expected stdout reflects the mutation. First confirm the current failure mode (the roadmap notes it miscompiles: by-value call / verifier failure).
- [ ] **Step 2: Run → FAIL** — confirm the exact current error (verifier failure or wrong output).
- [ ] **Step 3: Implement** — when a method has a pointer receiver and the call target is an addressable value (an lvalue), take its address (`&c`) automatically and pass the pointer, mirroring Go's auto-`&`. Use the existing lvalue-address path (the same one assignment uses) rather than loading the value.
- [ ] **Step 4: Run → GREEN** — the mutation is observed; `make verify` green (value-receiver methods unaffected). If this proves to need broad method-dispatch rework beyond a localized fix, report DONE_WITH_CONCERNS or BLOCKED rather than expanding scope.
- [ ] **Step 5: Commit** — `fix(codegen): auto-address-of for pointer-receiver methods on addressable values (P2-3)`.
