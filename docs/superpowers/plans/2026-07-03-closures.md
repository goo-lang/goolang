# Closures (Branch B) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Implement Branch B of the approved closures design (docs/superpowers/specs/2026-07-03-closures-design.md): func literals in full expression position with Go capture-by-variable semantics via always-heap promotion, plus the `go func(){...}()` idiom.

**Architecture:** (T1) Func literals parse (new `FuncLitNode`, its own productions — explicitly NOT a third consumer of the `g_func_signature_*` side-channel), typecheck, and emit as private `__goo_lit_<n>` functions with the env-first signature from Branch A (`codegen_get_funcval_call_type`); non-capturing literals ship first (env ignored), covering assignment/callback/return/immediate-invocation shapes. (T2) Capture: the checker's scope chain gains a function-boundary marker; identifier resolution crossing a boundary records the capture on the literal and marks the variable captured; captured variables' storage becomes a `goo_alloc` heap slot whose POINTER sits in the value table exactly where the alloca pointer sat (uniform load/store treatment — enclosing function and closures alike); the env struct holds slot pointers; closure prologues rebind captured names to slots loaded from env. Shared mutation, returned closures, param capture, recursion-via-capture all fall out. (T3) `go func(){...}()` for ZERO-ARG literals: a zero-arg closure's `{fn, env}` maps directly onto the runtime's `goo_go(thunk, box)` shape — spawn with (fn_ptr, env_ptr); arg-carrying go-literals recorded as follow-up (capture replaces args in practice).

**Tech Stack:** C23, LLVM-C, Bison (T1 grammar — REAL conflict risk this time). Parser + AST + checker + codegen. No runtime changes expected (T3 reuses goo_go).
