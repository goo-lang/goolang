# v1 Phase 3 — Useful Go Breadth (Implementation Plan)

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:subagent-driven-development. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Close the gaps that make idiomatic Go programs fail today: string concatenation, named return parameters, `defer`, and Go-standard composite literals (`[]T{...}` / `map[K]V{...}`). This turns "a subset" into "a useful subset."

**Scope note:** The roadmap's P3-6 (`int`/`uint` → i64) is **excluded** — it's the standalone `int→i64`/constant-coercion milestone (blast radius already mapped). This phase is P3-1…P3-5.

**Architecture:** Mostly additive type-check + codegen, ordered easiest-first so the `test-golden` oracle is exercised continuously and the grammar-heavy composite-literal work lands last (a block there doesn't strand the contained wins). Ordering: P3-3 (string concat) → P3-5 (named returns) → P3-4 (defer) → P3-1 (composite-literal type-check) → P3-2 (composite-literal lowering).

**Tech Stack:** C23, the Goo type checker + codegen, `src/parser/parser.y` (yacc grammar), the runtime (`goo_string_concat` already exists), Makefile golden runner.

**Spec:** `docs/2026-06-29-v1-roadmap.md` (Phase 3).

## Global Constraints

- **Build env:** `eval $(opam env --switch=default)` before any `make`; ignore the `parser.tab.c` recipe-override warnings.
- **Gates:** `make verify` (incl. `test-golden`) + `make test` (76/1) stay green; every existing golden case keeps passing. Local hooks gate commits/pushes; CI disabled.
- **Grammar changes (P3-1, P3-5) must NOT add reduce/reduce or shift/reduce conflicts** beyond what exists — rebuild the parser and compare the conflict count; if a change adds conflicts, that's a regression to resolve or escalate.
- `int` is i32 today (separate milestone) — keep fixtures small-valued.
- Goo-native slice literal `[1,2,3]` already works and must keep working — Go-standard `[]int{1,2,3}` is the NEW additional form, not a replacement.
- Commit discipline: conventional commits, `--no-gpg-sign`, stage only named files. TDD: RED probe first (golden case for positive features, negative probe for rejections).
- **Escalate, don't force:** P3-1/P3-2 (composite-literal grammar/lowering) and P3-4 (defer) are medium/low confidence. If a task needs grammar rework beyond a localized change, or broad codegen restructuring, report BLOCKED / DONE_WITH_CONCERNS with specifics rather than introducing grammar conflicts or fragile codegen.

### Task 2 (P3-5): Named return parameters

**Files:**
- Modify: `src/parser/parser.y` (`func_result:` ~line 455 — accept `(name Type, ...)` named-result form)
- Modify: `src/ast/`, `src/types/`, `src/codegen/function_codegen.c` (bind named results as zero-initialized locals in scope; a bare `return` returns their current values)
- Test: `examples/named_return_probe.goo` (+ expected)

- [ ] **Step 1: RED golden probe** — `func split(sum int) (x int, y int) { x = sum - 1; y = 1; return }` (bare return yields the named results). Caller prints `x, y`. Expected deterministic stdout.
- [ ] **Step 2: Run → FAIL** — named results don't parse / bare return doesn't yield them.
- [ ] **Step 3: Implement** — grammar: `func_result` accepts a parenthesized list of `ident Type` (named) in addition to the existing forms. Rebuild parser, confirm NO new conflicts. Type checker + codegen: register each named result as a zero-initialized local in the function scope; a bare `return` (no values) in a function with named results returns their current values (codegen reads the locals). Interaction with P1-3/P2-1 bare-return checks: a bare return is now valid when the function has named results — ensure those checks allow it.
- [ ] **Step 4: GREEN**; `make verify` green; conflict count unchanged.
- [ ] **Step 5: Commit** — `feat: named return parameters (P3-5)`.

### Task 3 (P3-4): `defer` — type-check acceptance + LIFO codegen

**Files:**
- Modify: `src/types/` (accept `AST_DEFER_STMT` — currently "Unknown statement type")
- Modify: `src/codegen/statement_codegen.c` (`codegen_generate_defer_stmt` — currently a stub) + `function_codegen.c` (emit deferred calls in LIFO order at every function-exit path)
- Test: `examples/defer_probe.goo` (+ expected)

- [ ] **Step 1: RED golden probe** — `defer fmt.Println("c"); defer fmt.Println("b"); fmt.Println("a")` → expect `a\nc\nb` (LIFO). Add a second case with `defer` + an early `return` to prove deferred calls run on that path too.
- [ ] **Step 2: Run → FAIL** — today: type error "Unknown statement type" (defer not accepted).
- [ ] **Step 3: Implement** — type-check: accept `defer <call>` (validate it's a call expression). Codegen: maintain a per-function stack of deferred call thunks; emit them in **reverse** (LIFO) order immediately before each `ret` (the normal exit and any early `return`). Scope: MVP supports `defer <fn>(args)` with args evaluated at defer-time (Go semantics) — if full arg-capture semantics prove broad, support the common no-arg / simple-arg case and note the limitation. Escalate if LIFO-at-every-exit needs broad return-path restructuring.
- [ ] **Step 4: GREEN** — LIFO order correct incl. early-return path; `make verify` green.
- [ ] **Step 5: Commit** — `feat(codegen): defer with LIFO execution at function exit (P3-4)`.

- Test: `examples/composite_slice_probe.goo`, `examples/composite_map_probe.goo` (+ expected)

- [ ] **Step 1: Investigate + RED golden probes** — `xs := []int{1,2,3}; fmt.Println(xs[0], xs[2])` → `1 3`; a `map[string]int{"a":1}` case. First determine WHY `[]int{...}` misparses (the lexer/grammar routes `[]Type{` to a struct literal `Type{...}`). Document the parse path before changing it.
- [ ] **Step 2: Run → FAIL** — "Unknown type 'int' in struct literal".

If disambiguating `[]T{` from struct-literal `T{` requires GLR/lexer-feedback beyond a localized grammar tweak, escalate with the specific conflict.
- [ ] **Step 4: GREEN at the type-check level** — the literals type-check (codegen may still be P3-2's job; if the existing slice-lit lowering already handles an explicit-element-type SliceLit, the probe may go fully green here — note it). `make verify` green; conflicts unchanged.
- [ ] **Step 5: Commit** — `feat: parse + type-check Go-standard []T{} / map[K]V{} composite literals (P3-1)`.
