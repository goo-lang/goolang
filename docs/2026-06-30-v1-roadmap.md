# Goo v1 Roadmap — evidence-based synthesis (refresh)

**Date:** 2026-06-30  
**Source:** `goo-v1-roadmap` workflow (run wf_7d8d8a4b-f3a) — 17 agents, ~1.1M tokens, every claim checked by running `bin/goo` (current tree, post F1–F8).  
**Supersedes:** `docs/2026-06-29-v1-roadmap.md` as the working v1 plan.

## v1 Definition

> Goo v1 = a compiler that reliably compiles a clean checkout, then compiles AND runs a useful Go-compatible subset (functions, structs, methods, interfaces, slices/maps, control flow, the common stdlib surface fmt/strings/math/os/strconv/errors) with Goo's two differentiators — error unions !T and nullable ?T — working END-TO-END on BOTH success and failure paths, including on methods and inside composite literals. v1 explicitly requires: comparisons and short-circuit evaluation that match Go (no silent miscompiles), runtime safety checks (bounds, divide-by-zero), a Go-like CLI (goo build/run with correct exit-code propagation), and ONE ccomp-free authoritative test gate whose pass rate reflects real source-to-executable behavior. Grand-vision items are out of scope.

## Verified baseline (real state)

Verified real state (not optimistic): The bin/goo pipeline (lex → parse → typecheck → LLVM IR → link) DOES produce runnable native executables today for a substantial Go subset — structs, value/pointer methods, multi-return, slices, maps, range, defer (non-loop), goroutines+channels, and BOTH differentiators on FREE functions all compile, link, and run correctly (65/65 golden probes pass). Adversarial checks REFUTED three load-bearing assumptions: (1) ASI is NOT the parse blocker — newline-separated statements already parse without explicit semicolons and the lexer already does targeted operator-continuation ASI; (2) empty/normal main DOES link and run; (3) structs and methods DO work end-to-end (only interfaces are absent). The TRUE foundational defect is that a clean `make` does NOT build the runtime archive as a compiler prerequisite, so the FIRST compile on a fresh checkout fails to link (the current tree only works because the archive is stale-prebuilt). Other verified gaps: interfaces are entirely unparseable (INTERFACE token has zero grammar rules) so all interface infra is dead; string and bool ==/!= have NO codegen (blocks a huge swath of Go); && / || do NOT short-circuit (correctness bug); no bounds or divide-by-zero checks (unsafe, silent wrong results); the !T catch ERROR path is dead (returns the value, ignores the handler) and !T METHODS fail to compile; nullable composite-literal fields/elements silently miscompile (value lost / nil read as 0); bitwise/shift precedence is wrong vs Go (silent miscompiles); fmt has only Println (no Printf — the flagship hello.goo does not compile); imports are a no-op and only 4 packages are hardcoded. The headline "100% test pass" measures type-system UNIT tests that never invoke bin/goo; the real e2e gate (make verify) cannot even run without CompCert/ccomp. The advanced subsystems (macros, async, security/taint, optimizer, auto-parallel, package manager) are standalone/dead in the pipeline; only comptime-const is genuinely wired (fib(10) folds to 55). The ~181KB stdlib/*.goo tree does not parse at all.

## Critical path

1. P0-1 Runtime archive as a build prerequisite — the true keystone: a clean `make` currently cannot link anything (refutes the prompt's ASI-keystone hypothesis; ASI already works for newline-separated statements).
1. P0-3 Convert the ASI newline handler from recursion to a loop — closes the only verified ASI defect (blank-line stack overflow / core dump).
1. P0-4 ccomp-free `make test-e2e` — without it there is no runnable e2e gate, so nothing downstream can be verified.
1. P1-1 string ==/!= codegen — unblocks a huge swath of real Go code; prerequisite for string ordering and string switch.
1. P1-5 && / || short-circuit — correctness keystone; guards every nil-check-then-deref idiom.
1. P2-1 → P2-2 catch handler typecheck then error-path codegen — makes !T (the headline differentiator) complete on its failure path end-to-end.
1. P2-3 method-aware error-union codegen — !T methods, currently a hard compile failure.
1. P3-1 Go precedence table + P3-2 trailing commas — correctness miscompile fix plus the single highest-leverage Go-source-compatibility fix.
1. P3-12 %expect/conflict pin — gate that lets interface grammar land safely.
1. P4-1 → P4-3 → P4-5 interface grammar → satisfaction typecheck → vtable dispatch — the largest missing Go feature; P4-5 is the biggest single unknown.
1. P5-1 fmt.Printf + P5-3 import validation — makes the flagship example compile and imports honest.
1. P6-3 goo build/run subcommands — the Go-like CLI v1 users expect.
1. P7-6 `make ci` single gate — the authoritative pass/fail that ties the whole v1 claim to verified e2e behavior.

## Open decisions

- Interface scope: are Go interfaces in v1 (Phase 4 is the single largest chunk, ~5-7 days incl. an unproven vtable-codegen path P4-5), or is v1 a struct/method-only Go subset with interfaces deferred? A 'Go-compatible subset' without interfaces is materially weaker, but P4-5 is the biggest risk.
- Closures/variadics depth: parser+typecheck acceptance (P3-5/P3-6) is cheap, but full codegen (captured variables, variadic lowering, append spread) is a separate large effort — decide whether v1 promises them runnable or just parseable.
- Expression-grammar: pin conflicts with %expect (cheap, Phase 3) vs the full 2-3 day reduce/reduce refactor. Recommend %expect for v1 unless a new feature is blocked by a reduce/reduce disable.
- fmt.%v reflection scope: how far does P5-6 go — scalars + error/.Error() + nullable only, or structs/slices/maps too? Full %v of composites is open-ended.
- Error construction model: implement fmt.Errorf (richer, more Go-faithful) for v1, or ship error()/errors.New only and document the limitation?
- Multi-file / user packages: confirm v1 is single-file + intrinsic-allowlist stdlib (recommended), explicitly excluding `import` of user code. This caps how 'useful' the subset is.
- Go ASI fidelity vs Goo superset: decide whether to fully match Go ASI for return/break and the (/[ boundary (P3-3/P3-4) or accept documented divergences, since the current behavior is a tolerant superset rather than strict Go.
- comptime const: prove it is load-bearing beyond plain const folding (recursive comptime fns) or fold the redundant call away and document equivalence.
- Stdlib aspirational tree: remove the ~181KB non-parsing stdlib/*.goo outright, or move under design/ as documented intent (P5-8) — confirm which before deleting.

## Post-v1 (explicitly out of scope, incl. self-hosting)

- Self-hosting / CompCert bootstrap pilot — decouple from the day-to-day e2e gate; revisit after v1 stabilizes.
- Interface advanced layers (concept-based generics, higher-kinded types, type-level programming, protocol-oriented programming) — the infra exists (interface_integration.c, concept_generics.c, protocol_oriented*.c) but is unreachable; wire into the working interface core only after basic Go interfaces ship.
- Macros (advanced_macro_system.c / derive_macros / template_macros) — linked but never invoked; wire expand_macro into parser/typecheck post-v1.
- Async runtime (src/async/*) — standalone demo only; integrate into the goroutine/channel core post-v1.
- Security / taint analysis (taint_analysis.c, src/security/*) — test-target only; wire into the type checker post-v1.
- Optimizer (advanced_optimization.c, profile_guided_optimization.c, comptime/optimization.c) — thread -O into codegen and call a real LLVM pass pipeline post-v1.
- Auto-parallelization (auto_parallel_analysis.c, dependency_analysis.c) — not compiled into bin/goo; wire post-v1.
- Package manager / multi-file & user packages (gmod, src/package/*) — currently un-buildable and the import system is a no-op; real module resolution is post-v1 (v1 ships an intrinsic-allowlist stdlib only).
- WASM and GPU targets — only reachable via internal self-tests with no --target/--emit flag; real backends post-v1.
- Full LSP/DAP backends, working REPL evaluation, time-travel debugging, hot reload, performance dashboards — ship as experimental in v1, build real backends post-v1.
- Formal verification, microkernel, capability security, Rust interop — grand-vision, untouched in v1.
- Expression-grammar refactor to eliminate the 219 reduce/reduce conflicts — only needed if the %expect band-aid proves insufficient; otherwise post-v1.
- Richer error values (fmt.Errorf, typed/wrapped errors), defer-in-loop, postfix ++ on s[i]/p.x, goroutines over closures/bound methods — incremental Go-completeness post-v1.
