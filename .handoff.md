# Handoff: 2026-07-05 — range over fixed arrays FIXED (branch `fix/range-over-arrays`)

`for range arr` over a fixed array was broken: it fell through to the slice/string
path (`extractvalue` field 0/1 of the raw `[N x T]` aggregate), so it misread element 0/1
as data-ptr/len → **silent miscompile** (`for range [3]int{10,20,30}` printed 20, Go: 3),
`for i, v := range arr` failed codegen ("Undefined identifier 'v'"), and `for range [1]int{42}`
**SIGSEGV'd the compiler**. Fix (`src/codegen/statement_codegen.c` `codegen_generate_for_stmt`,
one function): a TYPE_ARRAY branch that (a) skips the auto-load (arrays are a raw `[N x T]`
value, not a `{ptr,len}` struct), (b) takes the iteration count from the STATIC
`data.array.length`, (c) uses the array's own address as the element base — spilling an
rvalue array (inline `[3]int{...}` literal) into a temp alloca — and (d) wires
`elem_type` for the value-binding form. Golden **243/0** (+1 `range_array_probe`), tripwire
82/256 unchanged (no grammar touched), test 76/1. Verified go-identical across 10 shapes
(key-only, key-value, nested `[2][3]int`, array-of-struct, array-in-struct-field,
break/continue, index-only, zero-valued, the former-segfault `[1]int`) + a controller
value-binding+mutation probe.

**NEW grammar follow-up discovered:** an UNPARENTHESIZED `[`-prefixed composite literal
directly after `range` (or `if`/`for`) — `for _, v := range [3]int{1,2,3} {` — fails to
parse (the literal's `{` is swallowed as the for-body brace). Valid Go: Go disambiguates
`[`-prefixed literals (slices/arrays) because they can't be a bare `TypeName{...}`.
Workaround: parenthesize — `range ([3]int{1,2,3})`. Grammar-gated (tripwire); separate task.

- **Mechanism: int64 key slot + per-map `key_kind` tag, linear-scan equality, no hash**
  (Task 1 runtime, Task 2 codegen). Every map now carries a `key_kind` set once at
  creation (map literal, `make`) alongside its existing value-boxing metadata; every
  runtime entry point (`goo_map_set_sv`/`get_sv`/`delete_sv`, comma-ok) branches on it —
  `KEY_STRING` keeps the original `strcmp`-based equality (unchanged for existing string
  maps), every other kind (`KEY_INT`, `KEY_BOOL`, `KEY_CHAR`, `KEY_PTR`) compares the
  packed `int64_t` slot with `==`. No hashing was added — key lookup is still the
  pre-existing linear scan, just over a wider key domain.

  string (unchanged), any integer/uint width, `bool`, `char`/rune/byte (an integer kind),

  boxing on the key side for any admitted kind (unlike values, which box everything
  non-inline; keys have no boxed path at all in this cycle, see deferrals below).
- **String keys deliberately bypass `codegen_map_value_to_slot`** (the value-side boxing
  helper) — a raw `PtrToInt` on the `char*` is required so the runtime's `strcmp` path
  compares real string content, not two distinct heap addresses. This was the specific
  subtlety Task 2's reviewer confirmed: routing a string key through the boxing helper
  would silently break every existing string-keyed map (box-address equality instead of
  content equality).
- **Comparability gate at `type_from_ast`'s `AST_MAP_TYPE` case** (Task 3,
  `type_checker.c`): a key type is admitted iff `TYPE_STRING`, `type_is_integer`,

  now support, so there is no gap between "typechecks" and "the runtime can actually key
  on it." Two distinct rejection messages, by design (see deferrals #1/#2 below): a
  Go-comparable-but-not-yet-wired key type (float, struct, interface, array) gets "not yet
  supported in v1"; a permanently-non-comparable-in-Go key type (slice, map, func) gets

### Deferrals and findings — record prominently, read before extending this feature

1. **DEFERRED to a future cycle: boxed-key equality** — comparable structs, floats
   (`float32`/`float64`), interfaces, and arrays are all comparable in Go and rejected
   here ONLY because the v1 runtime has no boxed-key equality path yet (linear scan over
   an `int64` slot has nowhere to put > 8 bytes of key data, and float bit-pattern
   equality has its own NaN/±0 subtleties not addressed here). Adding them needs a boxed
   key slot (mirroring the value side's `goo_alloc`'d box) plus a real per-kind equality
   function instead of `==` on the raw slot — struct keys in particular must NOT be
   compared via `type_equals`/naive `memcmp`-of-struct-bytes (see finding #2 below;
   `type_equals` is confirmed struct-blind) — a real field-by-field or name-based
   identity check is required whenever this is picked up.
2. **Permanently rejected, matches Go**: slices, maps, and funcs are never comparable in
   Go and so are never valid map keys here either — this is not a v1 gap, it's the
   correct terminal answer.
3. **Enum keys rejected as "not comparable" is conservative-correct for v1, not
   necessarily final**: Goo enums are payload-bearing sum types (closer to Rust `enum`
   than a C-style scalar enum), so a blanket "not comparable" is the safe default given
   payload-carrying variants have no defined equality here. A future refinement could
   special-case unit-only (no-payload) enum types as deferred-comparable (same boxed-key
   cycle as #1), but that distinction was NOT built this task — scope was the four
   inline-key kinds plus string only.
4. **Wording inaccuracy, not a behavior bug**: channel-typed keys hit the same
   "not comparable" message as slices/maps/funcs, but Go actually considers channels
   comparable (identity equality, like pointers) — channels were simply never wired into
   the inline-key admission set this cycle (no `KEY_CHAN` kind, no runtime support), so
   the message should say "not yet supported" (deferred, like #1) rather than "not
   comparable" (permanent, like #2). Follow-up if channel keys are ever prioritized:
   either fix the message to match reality, or wire in real support (channels are
   already opaque pointers under the hood, so admission would likely be as cheap as the
   pointer-key case).
5. **Map iteration order is UNCHANGED** — still deterministic reverse-insertion order
   (pre-existing documented deviation from Go's runtime-randomized order), not something
   this branch touched or needed to touch; `map_key_range_probe` was written sum-based
   specifically to stay order-independent rather than accidentally re-asserting a
   specific order as a new contract.
6. **Pre-existing latent bug FIXED as part of Task 3** (see the gate description above):
   `type_check_index_expr`'s `TYPE_MAP` case previously returned the value type without
   ever checking the index expression against the map's key type — a wrong-typed index
   (e.g. a string literal against `map[int]int`) silently type-checked. Now rejects with
   "Map key type mismatch", pinned by the third case in `mapkey-reject-probe`.

## Generic map values: SHIPPED (previous branch, PR #110 — spec `docs/superpowers/specs/2026-07-04-func-map-values-design.md`)
- **Any value type** admitted at the map-value gate — the `type_checker.c` rejection that
  previously limited map values to integer/bool/char/pointer is gone. Keys stay
  **string-only** (unchanged, separately recorded gap).
- **Inline vs boxed rule** (one predicate, `map_value_is_inline(Type*)`): integer family,
  `bool`, `char`, pointers, and (free, since their LLVM reps are already opaque pointers)
  `map`/`chan` values stay in the existing `int64_t` slot via `PtrToInt`/`IntToPtr` —
  enables `map[string]map[string]int` with no boxing. Everything else (funcvals, strings,
  floats, structs, slices, arrays, interfaces, nullables, error unions) heap-boxes: the
  slot holds a pointer to a `goo_alloc`'d, `LLVMSizeOf`-sized (ABI size, not summed Goo
  field sizes — the `chan_padded_probe` lesson) copy of the value.
- **Zero-guard semantics**: a missing key (plain read or comma-ok miss) yields V's
  `zeroinitializer` without dereferencing a null slot — `goo_map_get_sv` returns `0` on
  miss, and the unbox helper checks `slot == 0` before loading. Concretely: missing
  dispatch-table entry → `{null,null}` funcval, and *calling* it hits the existing
  nil-func panic (`call of nil function`, Go-parity for "call through a missing map
  entry"); missing string entry → `{null,0}`, which must print as `""` (same
  representation as an uninitialized `var s string`, not a map-specific special case).
- **Go non-addressability enforced**: `&m[k]` rejects at typecheck
  (`cannot take the address of a map value (map values are not addressable)`); partial
  writes through a map index (`m[k].F = v`, `m[k][i] = v`) reject at the codegen lvalue
  backstop (`cannot assign through a map value (map values are not addressable; assign
  the whole value: m[k] = v)`). Whole-value forms (`m[k] = v`) stay legal.
- **Boxes leak by decision** — overwrite and delete never free the old box, consistent
  with closure envs and interface boxes (no GC yet; `goo_free` exists for when that day
  comes). This branch's Task 5 golden (`map_overwrite_probe`) pins that overwrite is
  correct *behavior* (fresh box per write, `len()` doesn't grow, latest value reads back)
  without asserting anything about the freed-vs-leaked old box — leaking is the recorded
  GC-era item, not a bug.
- Coverage: dispatch tables (`map[string]func(int) int`, closures + named funcs +
  overwrite + comma-ok), strings, floats, structs (value-copy semantics), nested maps,
  boxed-value range, and now (this task) overwrite/len pinning across both a string map
  and a func map.
