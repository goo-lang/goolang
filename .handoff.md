## Type assertions and type switches: SHIPPED (this branch — spec
`docs/superpowers/specs/2026-07-04-type-assertions-design.md`, plan
`docs/superpowers/plans/2026-07-04-type-assertions.md`, 4-task SDD plan complete)
- **All three forms land, concrete targets only**: `v, ok := x.(T)` (comma-ok, Task 2),
  `v := x.(T)` (single-return, panics on a miss, Task 2), and `switch v := x.(type) {
  case T1: … case Tn, Tm: … default: … }` (Task 3). `x.(SomeInterface)` and `case
  SomeInterface:` are both rejected (v1 scope — see deferral #1 below).
- **Mechanism: vtable-pointer identity, no RTTI** (`codegen_interface_assert_match` /
  `codegen_interface_assert_unbox`, `interface_codegen.c`, introduced Task 2 and reused
  verbatim by Task 3's type-switch desugar — the vtable compare is not reimplemented per
  case). The dynamic check is `x.vtable == &goo.vtable.<ConcreteType>.<Interface>` — sound
  for any interface with >=1 method (distinct method sets produce distinct thunk tables),
  unsound for a zero-method interface (every concrete type's vtable is an identical empty
  array — see deferral #1). The static gate is `type_interface_satisfied`, the same
  helper interface-satisfaction already used elsewhere in the compiler; comma-ok's zero
  guard mirrors `codegen_map_slot_to_value`'s branch/phi pattern (reviewer-confirmed real
  zero value on a miss, not garbage); the single-return miss goes through a real `panic`
  call (reviewer-confirmed, not UB).
- **Task 4 (this task) locks the static-reject diagnostics into the regression suite**:
  `typeassert-reject-probe` bundles (a) `x.(T)` on a non-interface operand → "operand is
  not an interface type"; (b) an impossible assertion (target doesn't implement the
  operand's interface) → "impossible type assertion"; (c) assert-to-INTERFACE target →
  "not supported in v1". `typeswitch-reject-probe` bundles (a) the same case type
  appearing twice anywhere in one switch → "duplicate case type"; (b) two `default`
  clauses → "multiple defaults in type switch". Both new probes PASS and are wired into
  `verify:`, alongside the pre-existing `typeassert-abort-probe` (Task 2, single-return
  panic) and `typeassert-emptyiface-reject-probe` / `typeswitch-emptyiface-reject-probe`
  (empty-interface fix, commit `230a767`) — five probes total covering this feature's
  reject surface.

### Deferrals and findings — record prominently, read before extending this feature

1. **DEFERRED to the RTTI cycle, joins assert-to-interface**: empty-interface
   (`interface{}`) discrimination and primitive/builtin case types (`case int:`, `case
   string:` — anything with no vtable) have no signal under vtable-ptr identity (every
   concrete type's vtable for a 0-method interface is an identical empty `[0 x ptr]`
   array, so the compare always matches the first candidate — a silent miscompile, not a
   clean failure, if left unguarded). Rejected cleanly at typecheck in v1
   (`typeassert-emptyiface-reject-probe` / `typeswitch-emptyiface-reject-probe`, commit
   `230a767`); real support needs a runtime type-descriptor tag, same cycle as
   assert-to-interface.
2. **Single-return panic message is STATIC-type-names-only**: `interface conversion: <I>
   is not <T>` names the STATIC operand interface and the STATIC target type — there is
   no dynamic-type name to print (no RTTI), so a miss against `var a Animal = X{}; a.(Y)`
   reports "Animal is not Y", not "X is not Y" the way Go's runtime panic does. Recorded
   as a v1 deviation, not a bug.
3. **TOP PAPERCUT — `if v, ok := x.(T); ok { … }` (the if-init-statement form) does NOT
   parse.** Confirmed directly this task: `if v, ok := a.(X); ok { … }` fails with a plain
   `syntax error` at the `ok` token. This is a PRE-EXISTING general grammar gap (the `if`
   production has no init-statement arm at all, for ANY init-statement — not something
   this branch introduced or could fix in scope), but it's THE idiomatic Go
   assertion-guard form, so its absence is the single biggest ergonomic gap this feature
   ships with. The correct workaround today is the two-line form (`v, ok := x.(T)` then
   `if ok { … }`), used throughout this branch's own goldens. Flagging prominently rather
   than letting it hide as a footnote.
4. **`type_equals` has a struct-blind-equality fallback** (`src/types/types.c`'s
   `type_equals` switch has no `case TYPE_STRUCT`, so it falls to `default: return 1;` —
   ANY two same-kind types with no dedicated case, including two DIFFERENT named structs,
   compare equal). Task 3's duplicate-case-type detection cannot use `type_equals` for
   this reason and instead added a local `type_switch_case_type_same` helper
   (`type_checker.c`) that compares by kind AND, when either side has a receiver name
   (i.e. is a named/struct-like type), by that name string — sidestepping `type_equals`
   entirely for the case that matters here. This is NOT a fix to the underlying bug:
   `type_equals` itself is still struct-blind, and it's a general-purpose helper — grep
   shows it backs comparisons in `channel_checker.c`, `constraint_inference.c`, and
   `protocol_oriented_programming.c` at minimum, none of which this branch audited.
   Recording as a TRACKED FOLLOW-UP, not fixed globally here — scope was the type-switch
   dup-case diagnostic only.
5. **Dead `AST_TYPE_ASSERT_EXPR` enum value** (`include/ast.h:53`) — a leftover from an
   earlier naming pass; no struct or grammar production ever produces it (the real node
   is `AST_TYPE_ASSERT`, added later in the same enum, with a comment noting the dead
   sibling). Cleanup chore, zero runtime impact — safe to delete whenever `ast.h` next
   gets touched.
6. **Bison baseline moved 81 -> 82 S/R** (Task 3, type switch) — JUSTIFIED, not a
   regression: `type_switch_guard` makes `primary_expr`'s existing `struct_lit: identifier
   LBRACE …` alternative reachable right after `SWITCH identifier`, colliding with the
   plain switch's own pre-existing `SWITCH expression LBRACE case_clause_list RBRACE`
   fallback on the same `identifier`-then-`LBRACE` lookahead. Verified empirically NOT
   isolated to one arm (removing either type-switch arm alone leaves the count at 82; only
   removing `type_switch_guard`'s reachability entirely restores 81), and the specific
   ambiguous sequence is proven LEXICALLY UNREACHABLE for either switch form (the M10
   bridge unconditionally rewrites the first depth-0 `{` after `SWITCH` to `LBRACE_BODY`,
   so a bare unparenthesized struct literal as a switch tag/guard operand is a parse
   error, not a silent misparse). Full derivation in
   `.claude/skills/goo-grammar/references/conflict-ledger.md` ("Corrected justification:
   type-assertions Task 3"). `scripts/grammar-tripwire.sh`'s `EXPECTED_SR` is now `82`.

## Pointer-concrete interface boxing: SHIPPED (previous branch, PR #112 — spec
`docs/superpowers/specs/2026-07-04-ptr-iface-boxing-design.md`, plan
`docs/superpowers/plans/2026-07-04-ptr-iface-boxing.md`)
- **C representation** (`codegen_interface_box`, interface_codegen.c): when the concrete
  is `TYPE_POINTER` to a nameable method-bearing type, the data slot holds the POINTER
  itself (Go's own representation) and the vtable is built against the POINTEE type — no
  `goo_alloc`, no copy, one less allocation (and one less pre-GC leak) per pointer boxing.
  The existing thunk logic (value receiver loads through `data`, pointer receiver passes
  `data` raw) was already correct for both shapes once this normalization landed; vtable
  "collision" between `Box` and `*Box` becomes intentional reuse. Value-boxed path is
  byte-for-byte unchanged. All boxing sinks funnel through this one helper (var-decl init,
  assignment, multi-assign, call args, map values), so the fix applies at every call site
  at once — confirmed directly this task via the four sink probes below, not just the
  goldens' var-decl site (commit `1036f9f`).
- **Thunk guard** (`build_thunk`, interface_codegen.c): defensive entry check — if a
  `TYPE_POINTER` concrete ever reaches the thunk builder un-normalized, it emits an
  internal error and returns NULL instead of re-creating the old verifier-failure/
  garbage-output symptom. Belt-and-braces against future direct callers of
  `codegen_interface_vtable`/`build_thunk` (commit `1036f9f`).
- **Gate lifted** (`type_interface_satisfied`'s embed fallback, type_checker.c): the
  `TYPE_STRUCT`-only restriction that shielded users from the miscompile by cleanly
  rejecting `*Outer` promoted-method satisfaction is gone — `*Outer` now satisfies
  interfaces via promoted methods for real, ordered strictly AFTER the representation fix
  landed and was golden-covered (lifting first would have converted a clean error into
  wrong output) (commit `f9ff711`).
- **Sink-probe verification (this task, temp dir, not golden'd)**: call-arg (`take(&b)`),
  map-value (`m["p"] = &b` aliasing vs `m["v"] = b` copy), and var-decl/reassign
  (`var g Getter = &b1; g = &b2; ...; g = b1`) probes all compiled with `bin/goo` and
  diffed byte-for-byte against `go run` on the identical program — all three matched
  exactly.
- **Deviation on record — nil pointer in interface**: `var p *Box; var g Getter = p;
  g.Get()` (value-receiver method through a nil `*Box`). Go panics cleanly
  (`invalid memory address or nil pointer dereference`). Goo: observed SIGSEGV, rc=139,
  no stdout/stderr — a hard crash rather than a clean panic, consistent with Goo's
  existing no-nil-check raw-pointer-deref behavior everywhere else in the compiler (not
  new to this change). Predicted verbatim in the design spec's "Recorded, not fixed"
  section and confirmed exactly as predicted. Pointer-receiver-through-nil-`*Box` (which
  Go runs successfully, since it never dereferences) remains UNTESTED — no probe exercises
  a pointer-receiver-only interface yet; open until that case arises.
- Queue item 1 (the #109-final-review-find pointer-concrete miscompile) is CLOSED; its
  shield note (the `TYPE_STRUCT` gate as user protection) is now obsolete — the gate
  itself is gone, replaced by the fix it was shielding.

## Generic map values: SHIPPED (previous branch, PR #110 — spec `docs/superpowers/specs/2026-07-04-func-map-values-design.md`)
- **Any value type** admitted at the map-value gate — the `type_checker.c` rejection that
  previously limited map values to integer/bool/char/pointer is gone. Keys stay
  **string-only** (unchanged, separately recorded gap).
- **Inline vs boxed rule** (one predicate, `map_value_is_inline(Type*)`): integer family,
  `bool`, `char`, pointers, and (free, since their LLVM reps are already opaque pointers)
  `map`/`chan` values stay in the existing `int64_t` slot via `PtrToInt`/`IntToPtr` —
  enables `map[string]map[string]int` with no boxing. Everything else (funcvals, strings,
  floats, structs, slices, arrays, interfaces, nullables, error unions) heap-boxes: the
  slot holds a pointer to a `goo_alloc`'d, `LLVMSizeOf`-sized (ABI size, not summed Goo
  field sizes — the `chan_padded_probe` lesson) copy of the value.
- **Zero-guard semantics**: a missing key (plain read or comma-ok miss) yields V's
  `zeroinitializer` without dereferencing a null slot — `goo_map_get_sv` returns `0` on
  miss, and the unbox helper checks `slot == 0` before loading. Concretely: missing
  dispatch-table entry → `{null,null}` funcval, and *calling* it hits the existing
  nil-func panic (`call of nil function`, Go-parity for "call through a missing map
  entry"); missing string entry → `{null,0}`, which must print as `""` (same
  representation as an uninitialized `var s string`, not a map-specific special case).
- **Go non-addressability enforced**: `&m[k]` rejects at typecheck
  (`cannot take the address of a map value (map values are not addressable)`); partial
  writes through a map index (`m[k].F = v`, `m[k][i] = v`) reject at the codegen lvalue
  backstop (`cannot assign through a map value (map values are not addressable; assign
  the whole value: m[k] = v)`). Whole-value forms (`m[k] = v`) stay legal.
- **Boxes leak by decision** — overwrite and delete never free the old box, consistent
  with closure envs and interface boxes (no GC yet; `goo_free` exists for when that day
  comes). This branch's Task 5 golden (`map_overwrite_probe`) pins that overwrite is
  correct *behavior* (fresh box per write, `len()` doesn't grow, latest value reads back)
  without asserting anything about the freed-vs-leaked old box — leaking is the recorded
  GC-era item, not a bug.
- Coverage: dispatch tables (`map[string]func(int) int`, closures + named funcs +
  overwrite + comma-ok), strings, floats, structs (value-copy semantics), nested maps,
  boxed-value range, and now (this task) overwrite/len pinning across both a string map
  and a func map.
