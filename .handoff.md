## Struct embedding: SHIPPED (this branch — spec `docs/superpowers/specs/2026-07-04-struct-embedding-design.md`)
- Value embedding (`struct { Base }`) and pointer embedding (`struct { *Base }`, promotion
  through an auto-deref hop) — both field and method promotion, any depth.
- Go-exact ambiguity semantics, including diamonds: shallowest-unique wins, outer shadows
  inner, same-depth collision is an error at the **use site only** (declaring the ambiguous
  struct is legal), explicit paths (`o.Base.X`) always work.
- Interface satisfaction through promoted methods: resolver fallback in
  `type_interface_satisfied` plus path-walking vtable thunks (`build_thunk` GEP-hops through
  embedded fields, including pointer-embedded hops, before dispatch). Task 6 shipped the
  value-embedding case golden-covered; this task closed the gap it flagged — pointer-embedding
  hop + pointer-receiver mutation observed through shared state is now golden-covered
  (`embed_iface_ptr_probe`: two interface-boxed values sharing one `*Base` see each other's
  `Bump()` mutations — go-run-verified identical).
- Refinement vs Go: a ONE-LINE `struct { Base }` needs an explicit `;` —
  `struct { Base; Z int }`. No bare-identifier production (would S/R-conflict with
  `identifier type` on IDENT/MULTIPLY lookahead); multi-line structs get `;` for free from
  the new struct-body-scoped ASI.
- Deviation (deliberate, documented in the spec): Go's method-set asymmetry (pointer-receiver
  methods excluded from `T`'s method set) is **not enforced** — Goo's dispatch is already
  compiler-wide lenient (auto-address/auto-deref), and embedding just inherits that leniency.
  Compatible-superset deviation, not a bug.
