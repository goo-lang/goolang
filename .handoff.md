# Handoff: 2026-07-05 — interface-typed map keys SHIPPED (branch `feat/interface-map-keys`) — map-key story COMPLETE

`map[InterfaceType]V` / `map[any]V` now work — **closes the map-key story** (string, int/bool/rune/byte,
pointer, non-string, struct, and now interface keys). Equality = dynamic-type identity + dynamic-value
equality; uncomparable dynamic types **runtime-panic** (Go-faithful, "comparing uncomparable"). Spec/plan
`docs/superpowers/{specs,plans}/2026-07-05-interface-map-keys*`.

Mechanism: each vtable now carries a per-concrete-type **value-equality fn at slot 0** (method thunks
shifted to slots 1..n; the one dispatch GEP is `idx+1`). A generic `goo_iface_key_eq` does vtable-identity
→ `vtable[0]` dispatch on the boxed `data` words. `GOO_MAPKEY_IFACE=3`; interface keys heap-copy `{vtable,
data}` (pointer in slot). Struct dynamic values reuse `goo.structeq.<T>` (#129); scalar/string deref-compare;
**pointer-boxed interfaces use pointer-identity eq** (`goo.ptreq`, via the `pointer_form` `$ptr$` vtable —
this also makes `nil` keys work, since `nil` is a null pointer → `null==null`). Also **added the `any`
type** (Goo lacked it) as a predeclared empty-interface alias, and a `codegen_box_map_key_if_needed` helper
boxing concrete key exprs into the interface at every key site.

Golden **265/0** (+7: `map_iface_key_probe`/`_string`/`_struct`/`_nil`/`_stringer`/`_range`/`_pointer`/
`_nil_direct`), `iface-map-key-uncomparable-probe` (runtime panic) in `verify:` ALL GREEN, ccomp PASS,
**bison 82/256** (no grammar change), full interface-method-dispatch/assertion/embedding suite unregressed
(the vtable shift's regression net).

### Findings / notes
- **Pre-existing (NOT this feature): Goo conflates `int` ≡ `int64`** (assigns int64→int without conversion;
  Go rejects). So `map[any]` with `int` vs `int64` keys collide — interface keys correctly *inherit* Goo's
  type identity. Genuinely-distinct types (int/string/struct/pointer/nil) are all go-identical.
- **DEFERRED (separate future item): interface-representation refactor → RTTI / type-switch-on-`any`.**
  `switch x.(type)` on an empty interface is still "not supported (requires runtime type information)". The
  vtable-slot-0 eq + `$ptr$` distinct-identity machinery is groundwork; a real type-descriptor (type_id +
  eq + methods) would unlock RTTI. NOT needed for map keys (done); its own design cycle. Timing nuance:
  uncomparable-key panic fires on first *compare* (linear scan), vs Go's insert-hash — both panic.
- **Known limitation (final-review Minor, non-blocking): an uncomparable dynamic key that NEVER
  same-type-compares is silently accepted** (`m[1]=1; var f any = func(){}; m[f]=2` → Goo exit 0, Go
  panics). `goo_iface_key_eq` short-circuits on the vtable mismatch before reaching the `goo.uncmpeq`
  stub. Only affects INVALID-Go programs (uncomparable keys always panic in Go); never a wrong value for
  valid code. A Go-faithful fix = an insert-time uncomparable check at `goo_map_set_sv`; deferred.

# Handoff: 2026-07-05 — grammar papercuts assessed + BLOCKED (lexer-bridge risk)

Two grammar-queue papercuts were assessed via the goo-grammar skill and deliberately NOT
shipped — both are lexer-bridge changes the grammar tripwire cannot guard (a bridge edit can
silently alter parse behavior without moving the 82 S/R + 256 R/R conflict counts), so a
speculative attempt risks silent parse regressions. These need dedicated lexer-bridge work,
not a papercut-sized change:

1. **Unparenthesized `[`-prefixed composite literal in a control clause** —
   `for _, v := range [3]int{1,2,3} {`, `if x := []int{...}; ...`. Root cause: the LBRACE_BODY
   lexer bridge (goo-grammar workaround #1) emits `LBRACE_BODY` for the first `{` after a
   control-clause condition, so the literal's `{` is swallowed as the loop/if body. Go
   disambiguates because `[`-prefixed types can't begin a block. Fix requires the bridge to
   recognize "the `{` right after a complete `[N]T`/`[]T` type is a literal, the NEXT `{` is
   the body" — parser-level context the lexer doesn't currently track. **Workaround: parenthesize —
   `range ([3]int{1,2,3})`** (already used in the range_array_probe golden). Real Go-compat gap;
   dedicated task.
2. **`}; stmt` and single-line `;`-separated statements in a block** (`for i<5 { a[i]=i; i++ }`).
   Rated rare/cosmetic (gofmt always splits statements onto separate lines); statement-separator
   grammar/lexer change, deferred.

Also still deferred (LARGE — new runtime machinery, not a papercut): interface-typed and
struct-typed map keys (need a boxed-key equality path; struct keys must NOT use struct-blind
`type_equals` — must synthesize a field-wise comparator). Minor: elided `{...}` composite for a
POINTER map-value type (`map[K]*P{k: {...}}`) not supported (use `&P{...}`).

# Handoff: 2026-07-05 — range over fixed arrays FIXED (branch `fix/range-over-arrays`)

`for range arr` over a fixed array was broken: it fell through to the slice/string
path (`extractvalue` field 0/1 of the raw `[N x T]` aggregate), so it misread element 0/1
as data-ptr/len → **silent miscompile** (`for range [3]int{10,20,30}` printed 20, Go: 3),

`data.array.length`, (c) uses the array's own address as the element base — spilling an
rvalue array (inline `[3]int{...}` literal) into a temp alloca — and (d) wires
`elem_type` for the value-binding form. Golden **243/0** (+1 `range_array_probe`), tripwire
82/256 unchanged (no grammar touched), test 76/1. Verified go-identical across 10 shapes
(key-only, key-value, nested `[2][3]int`, array-of-struct, array-in-struct-field,
break/continue, index-only, zero-valued, the former-segfault `[1]int`) + a controller
value-binding+mutation probe.

directly after `range` (or `if`/`for`) — `for _, v := range [3]int{1,2,3} {` — fails to

### Deferrals and findings — record prominently, read before extending this feature

1. **DEFERRED to a future cycle: boxed-key equality** — comparable structs, floats
   (`float32`/`float64`), interfaces, and arrays are all comparable in Go and rejected
   here ONLY because the v1 runtime has no boxed-key equality path yet (linear scan over
   an `int64` slot has nowhere to put > 8 bytes of key data, and float bit-pattern
   equality has its own NaN/±0 subtleties not addressed here). Adding them needs a boxed
   key slot (mirroring the value side's `goo_alloc`'d box) plus a real per-kind equality
   function instead of `==` on the raw slot — struct keys in particular must NOT be
   compared via `type_equals`/naive `memcmp`-of-struct-bytes (see finding #2 below;
   `type_equals` is confirmed struct-blind) — a real field-by-field or name-based
   identity check is required whenever this is picked up.
2. **Permanently rejected, matches Go**: slices, maps, and funcs are never comparable in
   Go and so are never valid map keys here either — this is not a v1 gap, it's the
   correct terminal answer.
3. **Enum keys rejected as "not comparable" is conservative-correct for v1, not
   necessarily final**: Goo enums are payload-bearing sum types (closer to Rust `enum`
   than a C-style scalar enum), so a blanket "not comparable" is the safe default given
   payload-carrying variants have no defined equality here. A future refinement could
   special-case unit-only (no-payload) enum types as deferred-comparable (same boxed-key
   cycle as #1), but that distinction was NOT built this task — scope was the four
   inline-key kinds plus string only.
4. **Wording inaccuracy, not a behavior bug**: channel-typed keys hit the same
   "not comparable" message as slices/maps/funcs, but Go actually considers channels
   comparable (identity equality, like pointers) — channels were simply never wired into
   the inline-key admission set this cycle (no `KEY_CHAN` kind, no runtime support), so
   the message should say "not yet supported" (deferred, like #1) rather than "not
   comparable" (permanent, like #2). Follow-up if channel keys are ever prioritized:
   either fix the message to match reality, or wire in real support (channels are
   already opaque pointers under the hood, so admission would likely be as cheap as the
   pointer-key case).
5. **Map iteration order is UNCHANGED** — still deterministic reverse-insertion order
   (pre-existing documented deviation from Go's runtime-randomized order), not something
   this branch touched or needed to touch; `map_key_range_probe` was written sum-based
   specifically to stay order-independent rather than accidentally re-asserting a
   specific order as a new contract.
6. **Pre-existing latent bug FIXED as part of Task 3** (see the gate description above):
   `type_check_index_expr`'s `TYPE_MAP` case previously returned the value type without
   ever checking the index expression against the map's key type — a wrong-typed index
   (e.g. a string literal against `map[int]int`) silently type-checked. Now rejects with
   "Map key type mismatch", pinned by the third case in `mapkey-reject-probe`.
