## Pointer-concrete interface boxing: SHIPPED (this branch — spec
`docs/superpowers/specs/2026-07-04-ptr-iface-boxing-design.md`, plan
`docs/superpowers/plans/2026-07-04-ptr-iface-boxing.md`)
- **C representation** (`codegen_interface_box`, interface_codegen.c): when the concrete
  is `TYPE_POINTER` to a nameable method-bearing type, the data slot holds the POINTER
  itself (Go's own representation) and the vtable is built against the POINTEE type — no
  `goo_alloc`, no copy, one less allocation (and one less pre-GC leak) per pointer boxing.
  The existing thunk logic (value receiver loads through `data`, pointer receiver passes
  `data` raw) was already correct for both shapes once this normalization landed; vtable
  "collision" between `Box` and `*Box` becomes intentional reuse. Value-boxed path is
  byte-for-byte unchanged. All boxing sinks funnel through this one helper (var-decl init,
  assignment, multi-assign, call args, map values), so the fix applies at every call site
  at once — confirmed directly this task via the four sink probes below, not just the
  goldens' var-decl site (commit `1036f9f`).
- **Thunk guard** (`build_thunk`, interface_codegen.c): defensive entry check — if a
  `TYPE_POINTER` concrete ever reaches the thunk builder un-normalized, it emits an
  internal error and returns NULL instead of re-creating the old verifier-failure/
  garbage-output symptom. Belt-and-braces against future direct callers of
  `codegen_interface_vtable`/`build_thunk` (commit `1036f9f`).
- **Gate lifted** (`type_interface_satisfied`'s embed fallback, type_checker.c): the
  `TYPE_STRUCT`-only restriction that shielded users from the miscompile by cleanly
  rejecting `*Outer` promoted-method satisfaction is gone — `*Outer` now satisfies
  interfaces via promoted methods for real, ordered strictly AFTER the representation fix
  landed and was golden-covered (lifting first would have converted a clean error into
  wrong output) (commit `f9ff711`).
- **Sink-probe verification (this task, temp dir, not golden'd)**: call-arg (`take(&b)`),
  map-value (`m["p"] = &b` aliasing vs `m["v"] = b` copy), and var-decl/reassign
  (`var g Getter = &b1; g = &b2; ...; g = b1`) probes all compiled with `bin/goo` and
  diffed byte-for-byte against `go run` on the identical program — all three matched
  exactly.
- **Deviation on record — nil pointer in interface**: `var p *Box; var g Getter = p;
  g.Get()` (value-receiver method through a nil `*Box`). Go panics cleanly
  (`invalid memory address or nil pointer dereference`). Goo: observed SIGSEGV, rc=139,
  no stdout/stderr — a hard crash rather than a clean panic, consistent with Goo's
  existing no-nil-check raw-pointer-deref behavior everywhere else in the compiler (not
  new to this change). Predicted verbatim in the design spec's "Recorded, not fixed"
  section and confirmed exactly as predicted. Pointer-receiver-through-nil-`*Box` (which
  Go runs successfully, since it never dereferences) remains UNTESTED — no probe exercises
  a pointer-receiver-only interface yet; open until that case arises.
- Queue item 1 (the #109-final-review-find pointer-concrete miscompile) is CLOSED; its
  shield note (the `TYPE_STRUCT` gate as user protection) is now obsolete — the gate
  itself is gone, replaced by the fix it was shielding.

## Generic map values: SHIPPED (previous branch, PR #110 — spec `docs/superpowers/specs/2026-07-04-func-map-values-design.md`)
- **Any value type** admitted at the map-value gate — the `type_checker.c` rejection that
  previously limited map values to integer/bool/char/pointer is gone. Keys stay
  **string-only** (unchanged, separately recorded gap).
- **Inline vs boxed rule** (one predicate, `map_value_is_inline(Type*)`): integer family,
  `bool`, `char`, pointers, and (free, since their LLVM reps are already opaque pointers)
  `map`/`chan` values stay in the existing `int64_t` slot via `PtrToInt`/`IntToPtr` —
  enables `map[string]map[string]int` with no boxing. Everything else (funcvals, strings,
  floats, structs, slices, arrays, interfaces, nullables, error unions) heap-boxes: the
  slot holds a pointer to a `goo_alloc`'d, `LLVMSizeOf`-sized (ABI size, not summed Goo
  field sizes — the `chan_padded_probe` lesson) copy of the value.
- **Zero-guard semantics**: a missing key (plain read or comma-ok miss) yields V's
  `zeroinitializer` without dereferencing a null slot — `goo_map_get_sv` returns `0` on
  miss, and the unbox helper checks `slot == 0` before loading. Concretely: missing
  dispatch-table entry → `{null,null}` funcval, and *calling* it hits the existing
  nil-func panic (`call of nil function`, Go-parity for "call through a missing map
  entry"); missing string entry → `{null,0}`, which must print as `""` (same
  representation as an uninitialized `var s string`, not a map-specific special case).
- **Go non-addressability enforced**: `&m[k]` rejects at typecheck
  (`cannot take the address of a map value (map values are not addressable)`); partial
  writes through a map index (`m[k].F = v`, `m[k][i] = v`) reject at the codegen lvalue
  backstop (`cannot assign through a map value (map values are not addressable; assign
  the whole value: m[k] = v)`). Whole-value forms (`m[k] = v`) stay legal.
- **Boxes leak by decision** — overwrite and delete never free the old box, consistent
  with closure envs and interface boxes (no GC yet; `goo_free` exists for when that day
  comes). This branch's Task 5 golden (`map_overwrite_probe`) pins that overwrite is
  correct *behavior* (fresh box per write, `len()` doesn't grow, latest value reads back)
  without asserting anything about the freed-vs-leaked old box — leaking is the recorded
  GC-era item, not a bug.
- Coverage: dispatch tables (`map[string]func(int) int`, closures + named funcs +
  overwrite + comma-ok), strings, floats, structs (value-copy semantics), nested maps,
  boxed-value range, and now (this task) overwrite/len pinning across both a string map
  and a func map.
